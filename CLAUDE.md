This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

### Frontend (`frontend/`)
```bash
npm run dev        # dev server on :5173
npm run build      # tsc --noEmit + vite build (always run before committing)
npm run test       # vitest run
```

### Backend (`backend/`)
```bash
npm run dev        # tsx watch (Supabase)
npm run dev:local  # tsx watch --env-file=.env.local (SQLite local)
npm run build      # tsc (outputs to dist/, copies sqlite-schema.sql)
npm run test       # vitest run
```

### Running locally end-to-end
Backend requires `USE_SQLITE=true` + `JWT_SECRET` in `.env.local`. Frontend requires `VITE_API_URL=http://localhost:3000` in `.env.local`.

## Architecture

### Two separate apps
- **`frontend/`** — React 18 + TypeScript + Vite SPA. No SSR. Deployed to Vercel as a static site.
- **`backend/`** — Express 4 + TypeScript API. Deployed to Vercel as serverless functions (`backend/vercel.json` + `backend/src/api/index.ts`).

### Backend database layer
`backend/src/supabase.ts` exports a single `supabase` client. When `USE_SQLITE=true`, it swaps in `backend/src/db/sqlite-adapter.ts` which mimics the Supabase JS client interface (`.from().select().eq()...`) against a local SQLite file. This lets the full API run locally without a Supabase account.

Schema lives in `backend/src/db/sqlite-schema.sql`. Production uses Supabase PostgreSQL — always keep the schema in sync.

### Auth
JWT-based. `POST /auth/login` returns a short-lived token. Two middleware guards in `backend/src/middleware/moderator.ts`: `requireModerator` (any logged-in mod) and `requireMaster` (role `master` only). The frontend stores the session in React state (no localStorage persistence).

### Scoring & code decoding
Players submit a base64 PZR code generated by the in-game Lua mod. `backend/src/lib/decoder.ts` (and its frontend mirror `frontend/src/lib/decoder.ts`) parse the code. `backend/src/lib/scoring.ts` computes the score from decoded fields + objectives.

### Frontend routing
React Router v6 with three routes in `frontend/src/App.tsx`:
- `/` → `MainView` (public rank)
- `/painel` → `PainelPage` (moderator panel, behind login)
- `/player/:id` → `PlayerPage` (public player profile)
- `/overlay/:id` → `OverlayPage` (stream overlay)

### Moderator panel (`PainelPage`)
Currently has two tabs (`Tab = 'players' | 'rank'`). The "Jogadores" tab renders `PendingPlayers`; the "Moderadores" tab renders `ModeratorsList`. Rank entries are rendered inline below these tabs (not yet a tab). All moderator API calls go through `frontend/src/lib/api.ts`.

### Trait system
Traits arrive from the DB as a CSV string in `entry.traits` (e.g. `"base:athletic,base:out of shape"`). `frontend/src/lib/traits.ts` exports:
- `TRAITS: Record<string, TraitDef>` — map of PascalCase → `{ name, type, image }`
- `resolveTrait(id)` — normalises `base:xxx` format and strips spaces before lookup
- `getTraitImageUrl(def)` — resolves `import.meta.glob` URL for the image file

Trait images live in `frontend/assets/caracteristicas/Positivos/` and `Negativos/`. The image file name must exactly match the `image` field in `TRAITS` (without `.png`).

### Objectives
`frontend/src/lib/objectives.ts` defines `SPIFFOS_RESTAURANTS` (list of Spiffo's locations by `id` and `name`) and `BASE_ITEMS`. The `id` field is the DB key inside `entry.objectives.bases`; never change an `id` after data exists in production.

## Key conventions

- **Version tags**: commits use `(vX.Y.Z)` suffix in message; every feature/fix gets a `git tag vX.Y.Z` + GitHub release via `gh release create`.
- **CSS**: all styles in `frontend/css/style.css`. Uses CSS custom properties (`--green`, `--red`, `--border`, etc.). Tailwind v4 tokens defined via `@theme` at the top. Minimum font size for readable text: **16px**.
- **Breakpoints**: `> 860px` table layout, `≤ 860px` card layout, `≤ 460px` single-column.
- **Player soft-state**: `blocked` = cannot receive new entries; `status = 'rejected'` = not in rank. These are separate fields. A soft-deleted player pattern does not yet exist in the schema — adding it requires a migration on both SQLite schema and Supabase.
